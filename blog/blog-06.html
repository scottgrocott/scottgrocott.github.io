The Great Engine Shootout

One JSON to rule them all: A-Frame vs. Three.js vs. Babylon.js

Posted by Scott Grocott on January 19, 2026

In my previous post, I discussed breaking the Metal Throne hub into modular assets and storing their coordinates in a JSON file. But there was a hidden motive behind this data-driven approach. By decoupling the "where" and "what" of my models from the "how" of the rendering, I’ve unlocked a new phase of development: Parallel Prototyping.

The Strategy: Parallel Development

I’ve decided to launch a "Shootout" between the three titans of web-based 3D: A-Frame, Three.js, and Babylon.js. Because my world exists as a JSON manifest, I can pull the exact same scene into all three environments simultaneously. I want to see which engine handles the industrial grit, the lighting, and the machine logic of Metal Throne with the most efficiency.

The Unified Data

This simple JSON structure acts as the source of truth. Every engine reads this same file to determine where the steel goes.

{
  "hub_elements": [
    {
      "name": "Heavy_Silo_01",
      "file": "silo.glb",
      "position": {"x": -10, "y": 0, "z": -20},
      "rotation": {"y": 45}
    }
  ]
}

The Contenders: Loading the Scene

1. A-Frame (The ECS Specialist)

A-Frame is fantastic for rapid entity management. It wraps the logic in a custom component that iterates through our data.

// A-Frame: Appending entities to the DOM
data.hub_elements.forEach(item => {
  const entity = document.createElement('a-entity');
  entity.setAttribute('gltf-model', item.file);
  entity.setAttribute('position', item.position);
  scene.appendChild(entity);
});
2. Three.js (The Native Powerhouse)

Going "pure" Three.js gives us granular control over the renderer, though it requires more manual scene management.

// Three.js: Direct GLTFLoader integration
data.hub_elements.forEach(item => {
  loader.load(item.file, (gltf) => {
    gltf.scene.position.set(item.position.x, item.position.y, item.position.z);
    scene.add(gltf.scene);
  });
});
3. Babylon.js (The Robust Engine)

Babylon offers a high-level API with professional-grade tools built-in, making it a strong contender for the complex industrial lighting I need.

// Babylon.js: SceneLoader approach
data.hub_elements.forEach(item => {
  BABYLON.SceneLoader.ImportMesh("", "", item.file, scene, (meshes) => {
    meshes[0].position = new BABYLON.Vector3(item.position.x, 0, item.position.z);
  });
});
"The engine shouldn't dictate the world; the world data should dictate the engine."

Looking Toward 2027

This experiment isn't just about the web. This shootout will continue throughout 2026 as I refine the mechanics of the Metal Throne. I’ll be comparing performance metrics, VR compatibility, and ease of development across these three libraries.

However, the web is only the beginning. This entire process—building modular assets and data-driven scenes—is preparation for **2027**, when I plan to migrate the project into **Godot**. By then, the logic and the layout will be so well-defined that the jump to a full-scale desktop engine will be a tactical evolution, not a starting over.

Which engine will claim the Throne? Stay tuned.
