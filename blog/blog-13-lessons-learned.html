Drone Wars: Lessons Learned

What A-Frame Taught Me About the Road Ahead

Posted by Scott Grocott on January 28, 2026

Drone Wars is functionally complete. Autonomous drones patrol flight zones, spatial audio tracks their movement, and the integration layer synchronizes three independent libraries into a coherent system. But the real value of this prototype isn't the working code‚Äîit's the lessons learned that will shape Go-Cart Wars, Forklift Wars, and ultimately Metal Throne: Fuel Yard.

What Worked: The Component Architecture

A-Frame's component-based architecture forced good habits. Each system‚Äînavigation, physics, audio‚Äîexists as an independent component that can be mixed and matched. This modularity will port beautifully to Three.js and Babylon.js.

// A-Frame component
AFRAME.registerComponent('spatial-sound', {
  schema: { src: {type: 'string'}, volume: {type: 'number'} },
  init: function() { /* setup */ },
  tick: function() { /* update */ },
  remove: function() { /* cleanup */ }
});

// Will translate to Three.js class
class SpatialSound {
  constructor(options) { /* setup */ }
  update(deltaTime) { /* update */ }
  dispose() { /* cleanup */ }
}

The patterns are identical. A-Frame simply provides the infrastructure.

What Didn't Work: The Abstraction Tax

A-Frame's convenience comes with cost. Simple operations require verbose code:

// A-Frame: Create a colored sphere
const sphere = document.createElement('a-sphere');
sphere.setAttribute('color', '#ff0000');
sphere.setAttribute('radius', '2');
sphere.setAttribute('material', 'shader: flat');
droneContainer.appendChild(sphere);

// Three.js equivalent (what I'll use in Go-Cart Wars)
const geometry = new THREE.SphereGeometry(2);
const material = new THREE.MeshBasicMaterial({color: 0xff0000});
const sphere = new THREE.Mesh(geometry, material);
droneContainer.add(sphere);

The Three.js version is more direct and gives finer control. For Go-Cart Wars, I'll work at the Three.js level while maintaining the component structure.

The Big Lesson: Authoring Over Automation

The failed navmesh experiment taught the most valuable lesson: automated solutions aren't always better than hand-crafted ones. This principle will extend beyond navigation:

Combat Encounters

Instead of procedurally spawning enemies, design encounters. Place spawn points, define patrol patterns, choreograph difficulty curves.

Audio Design

Instead of ambient sound generators, compose specific soundscapes for specific areas. The player's entrance to the Fuel Yard should sound intentional, not random.

Lighting

Instead of global illumination, hand-place dramatic lights. Babylon.js will excel here‚Äîits light system rewards artistic intent.

Rapier Physics: The Keeper

Rapier proved itself. It's fast, reliable, and the WebAssembly implementation means it'll work identically across all three projects. The API is pleasant:

// Same code for A-Frame, Three.js, and Babylon.js
const rigidBodyDesc = RAPIER.RigidBodyDesc.dynamic()
  .setTranslation(x, y, z);
const rigidBody = physicsWorld.createRigidBody(rigidBodyDesc);

const colliderDesc = RAPIER.ColliderDesc.ball(radius)
  .setMass(mass);
const collider = physicsWorld.createCollider(colliderDesc, rigidBody);

This consistency is invaluable. Go-Cart Wars and Forklift Wars will use the exact same physics code.

Yuka AI: Room for Growth

Yuka handled basic navigation well, but it's designed for ground-based agents. For Go-Cart Wars (ground vehicles) it'll be perfect. For Forklift Wars (complex interactions), I may need to extend or replace it.

The flight zone concept, though, is universal. Whether it's go-cart race lines or forklift work areas, defining regions of valid movement is cleaner than automatic pathfinding.

Howler Audio: Spatially Aware

Howler's spatial audio transformed the experience. Hearing drones approach from behind, tracking them by sound when they leave view‚Äîthis is essential for immersion. Go-Cart Wars will benefit even more, with engine sounds, tire screeches, and collision impacts all spatially positioned.

The Technology Triangle

The three-project approach is vindicated:

Project Engine Focus Tech Validated
Drone Wars A-Frame AI Navigation Yuka, Rapier, Howler integration
Go-Cart Wars Three.js Vehicle Physics Rapier vehicles, direct rendering control
Forklift Wars Babylon.js Complex Interactions Full engine features, professional pipeline

Each project builds on the previous. By the time Fuel Yard begins, I'll have production-tested all core systems.

Unexpected Discoveries

Debugging is Visual

Being able to see flight zones as green boxes was transformational. Future projects will prioritize visual debugging from day one. Three.js will get debug wireframes, Babylon.js will get the Inspector.

Performance Budgets Matter

With 15 drones, the frame rate dropped to 45 FPS. Not because of physics or AI‚Äîbecause of rendering. Each drone had a sphere (geometry), a light (shadow calculations), and text (texture generation). The lesson: visual complexity costs more than system complexity.

What's Next: Go-Cart Wars

With Drone Wars complete, development shifts to Go-Cart Wars‚Äîa Three.js prototype focused on vehicle physics and ground-based combat. The challenges ahead:

  • Rapier vehicle constraints: Wheels, steering, suspension
  • Terrain interaction: Bumps, jumps, friction variation
  • Competitive AI: Go-carts that race, not just patrol
  • Direct Three.js rendering: No A-Frame abstraction

The flight zone concept will translate to race lines. The component architecture will become ES6 classes. The spatial audio will expand to engine RPM mapping. And Rapier physics will finally tackle the complexity it was designed for: realistic vehicle simulation.

The Road to the Fuel Yard

"Three prototypes, three technologies, one throne."

Metal Throne: Fuel Yard will be the culmination: Babylon.js for professional visuals, Rapier for authentic physics, Yuka for intelligent enemies, Howler for immersive audio. But it won't be built from scratch‚Äîit'll be assembled from battle-tested components proven across three smaller games.

Drone Wars taught me that the path to a large project is through small, focused experiments. Go-Cart Wars will teach me vehicle dynamics. Forklift Wars will teach me Babylon.js mastery. And when the Fuel Yard finally opens, it will stand on a foundation of three successful prototypes.

The first battle is won. Two more await.
