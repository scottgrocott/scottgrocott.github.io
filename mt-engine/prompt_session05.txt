### Problem being worked on 🔧
**Player falls through heightfield terrain after applying heightmap in editor**

Diagnosis so far:
- Flat terrain collider works perfectly (player sits at Y=1.24)
- Heightfield collider IS created correctly (verified via collider dump)
- Player body IS disabled during freecam rebuild (verified via isEnabled())
- After freecam exit, player gets snapped to terrainY+height+0.5 = ~14.2
- Then falls through heightfield — heightfield doesn't stop the capsule

Current approach: Y-floor rescue in _tickWalk — if player.y < terrainFloor + height*0.5, 
snap back up. This is a safety net while the real heightfield collision bug is investigated.

### Suspected root cause
Rapier heightfield collision with capsule shape may require:
1. CCD (Continuous Collision Detection) enabled on player body
2. Different heightfield winding/orientation
3. heightfield scale params may need total size not per-cell size

### Physics architecture
```
Rapier world
├── fixed body → heightfield/cuboid collider  (terrain)
├── dynamic body → capsule collider           (player)  
├── kinematic body → sphere collider          (enemies - drones)
└── kinematic body → box collider             (enemies - cars/forklifts)
```

### Heightfield params
- subdiv = 64 → 65x65 sample grid
- scale = { x: sizeX/64, y: 1.0, z: sizeZ/64 }  (per-cell spacing)
- For 700x700 terrain: scale.x = 700/64 = 10.9375
- No translation (centered at origin matches Babylon ground mesh)
- Heights: Float32Array, row-major, no V-flip
